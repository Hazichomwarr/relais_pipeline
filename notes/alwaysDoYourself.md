# alwaysDoYourself.md

> A list of things I should ALWAYS build, think through, or deeply understand myself — even in the AI era.

---

# 1. Architecture Decisions

AI can suggest structures.

But I should decide:

- folder organization
- data flow
- separation of concerns
- scalability direction
- abstraction boundaries

Reason:
This develops engineering judgment.

---

# 2. Database Design

Always think through:

- tables
- relationships
- constraints
- indexing
- normalization
- access patterns

Reason:
Bad database decisions become expensive later.

---

# 3. State Management Logic

I should understand:

- what state exists
- where it lives
- who owns it
- when it updates
- derived vs source state

Reason:
This is core frontend engineering.

---

# 4. Business Logic

Always deeply understand:

- pricing logic
- workflows
- permissions
- validations
- automation rules
- edge cases

Reason:
Business logic IS the product.

Never outsource understanding of it.

---

# 5. Authentication & Authorization

I should understand:

- sessions
- JWTs
- cookies
- middleware
- protected routes
- roles/permissions

Reason:
Security matters.

Blindly trusting AI here is dangerous.

---

# 6. API Design

I should design:

- endpoints
- payload structure
- status handling
- error patterns
- contracts between frontend/backend

Reason:
Good APIs create maintainable systems.

---

# 7. Debugging

NEVER skip debugging myself.

Always investigate:

- why something broke
- stack traces
- logs
- browser behavior
- network requests

Reason:
Debugging builds true engineering intuition.

---

# 8. Performance Thinking

I should understand:

- unnecessary re-renders
- slow queries
- image optimization
- lazy loading
- caching
- bundle size

Reason:
Performance awareness separates builders from engineers.

---

# 9. UX Decisions

Always think through:

- user friction
- mobile experience
- clarity
- CTA placement
- loading states
- empty states
- form simplicity

Reason:
Users feel UX more than code quality.

---

# 10. Naming

Always carefully name:

- variables
- functions
- components
- database fields
- folders

Reason:
Naming reflects understanding.

---

# 11. Product Decisions

AI cannot fully decide:

- what features matter
- what to remove
- what users actually need
- what creates business value

Reason:
This is founder territory.

---

# 12. Final Refactoring Pass

Even if AI generated the code:

- clean it
- simplify it
- rename things
- remove unnecessary complexity

Reason:
Refactoring builds craftsmanship.

---

# 13. Reading Every Line Before Shipping

Never deploy code I don't understand.

Reason:
Fast shipping is good.
Blind shipping is dangerous.

---

# 14. Client Communication

Always personally handle:

- expectations
- scope
- pricing
- trust
- negotiation
- delivery communication

Reason:
Business survives on relationships.

---

# 15. Learning Core Fundamentals

Always continue strengthening:

- JavaScript/TypeScript
- React fundamentals
- networking basics
- databases
- browser behavior
- async flow
- HTTP
- Git

Reason:
Tools evolve.
Fundamentals compound forever.

---

# THINGS AI IS GREAT FOR

Use AI aggressively for:

- boilerplate
- repetitive UI
- scaffolding
- form generation
- Tailwind styling
- documentation
- regex
- debugging assistance
- idea exploration
- refactoring suggestions

Reason:
Leverage matters.

---

# GOLDEN RULE

Never ask:
"Did I write every line myself?"

Ask:
"Do I deeply understand the system I'm building?"

That is what matters.
